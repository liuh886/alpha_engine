# Structured 10D Research Paradigm

## Purpose

This feature turns the existing fixed-10D research assumptions into a
validated, serializable preparation contract.

It deliberately does **not** add another model runner. The first version stops
at:

```text
YAML spec
  -> validated ResearchParadigmSpec
  -> factor/candidate manifests
  -> standard research-run artifact paths
  -> Qlib-free frontend preparation payload
```

Real model execution remains outside this PR until the runner can prove that
the declared spec and effective runtime inputs are identical.

## Relationship to PR #92

PR #92 defines the replacement-first convergence direction:

```text
ResearchContract
  -> SignalFrame
  -> PortfolioIntent
  -> EvaluationReport
  -> EvidenceBundle
  -> PromotionDecision
  -> thin adapters
```

This PR implements only the earliest contract/preparation portion. It does not
claim to complete `PortfolioIntent`, evidence unification, or promotion
decision migration.

The structured paradigm is therefore a compatibility bridge toward
`ResearchContract`, not a second permanent architecture.

## What this PR replaces

It replaces ad-hoc duplication of:

- market and benchmark selection;
- universe source and alignment mode;
- factor-group selection;
- ranker calibration declarations;
- fixed 10D strategy assumptions;
- walk-forward preparation dates;
- metric lists and duplicated gate thresholds;
- artifact filenames.

The YAML refers to the canonical gate profile
`ten_day_model_gates_v1`; numeric thresholds remain owned by
`src/research/ten_day_model_gates.py`.

## What this PR does not do

- It does not initialize Qlib.
- It does not train or score a model.
- It does not call `run_cn_10d_validation.py`.
- It does not call `run_cn_feature_quality_validation.py`.
- It does not copy evidence from another experiment.
- It does not produce `data_readiness.json`,
  `walk_forward_stability.json`, or a model decision pack.
- It does not claim that any candidate is trade-ready.
- It does not connect a broker or create orders.

This boundary prevents a dangerous state in which `experiment_spec.json`
describes one candidate grid while the copied evidence was generated by a
different frozen runner.

## Public API

```python
from src.research.paradigm import (
    load_research_paradigm_spec,
    run_research_paradigm,
)

spec = load_research_paradigm_spec(
    "configs/research_paradigms/cn_10d_csi300_baseline.yaml"
)
result = run_research_paradigm(spec, dry_run=True)
```

`dry_run=False` fails closed.

## CLI

```bash
python scripts/run_research_paradigm.py \
  --spec configs/research_paradigms/cn_10d_csi300_baseline.yaml \
  --dry-run
```

The CLI has no execution-dispatch flag.

## Validation contract

The validator enforces:

- safe `experiment_id` slug;
- market in `cn` or `us`;
- universe market key equal to top-level market;
- existing universe and factor-library sources;
- unique selected factor groups;
- non-empty, unique calibration settings;
- positive `top_n` and `bottom_n`;
- fixed 10-day horizon, holding, rebalance, and embargo;
- canonical raw 10-day forward-return expression and provenance;
- valid requested training and test dates;
- at least three walk-forward windows;
- `benchmark_mode: reference_only`;
- exact canonical metric ordering;
- `gate_profile: ten_day_model_gates_v1`;
- `artifact_profile: research_run_v1`.

The spec cannot embed a duplicate numeric `gates` mapping.

## Prepared artifacts

A dry run writes:

```text
artifacts/research_runs/{experiment_id}/
  experiment_spec.json
  run_status.json
  factor_manifest.json
  candidate_manifest.json
  signals_latest.json
  top_bottom_signals.csv
  frontend_payload.json
```

It intentionally does not create empty files that look like real evidence.

`artifact_paths` lists only files that exist, except the frontend payload may
include its own target path while being written.

## Factor expression versioning

The CN YAML corrects parenthesization for risk-adjusted and price-volume
interaction expressions. Corrected factors use `:v2` ids, so evidence from the
legacy #91 expressions cannot be silently mixed with the corrected semantics.

## Next implementation step

After this PR, implement a spec-bound execution adapter in a separate PR.

That adapter must:

1. accept the selected feature groups, calibrations, baselines, universe,
   dates, alignment mode, and top/bottom settings from the validated spec;
2. emit an effective-input manifest;
3. prove in tests that declared and effective candidate manifests are equal;
4. generate evidence directly into the same run directory;
5. fail closed if required evidence artifacts are missing;
6. derive trade readiness only from a versioned promotion decision.

Until those conditions are met, the structured paradigm remains preparation
only.
